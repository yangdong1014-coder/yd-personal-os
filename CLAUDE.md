## 0 · 最高铁律（违反即停，无例外）

```
【修改路径铁律】严格单向，不可跳步、不可逆序：

  本地修改 → 本地验证通过 → 杨栋人工确认 → 推送 GitHub → 同步云端

  你（执行 AI）的权限边界，止于「本地修改 + 本地验证」。
  超出此边界的两步，你一律不得自动执行：

  ✗ 不得执行 git push（推 GitHub）
  ✗ 不得执行任何云端部署 / 同步 / 远程操作命令
  ✗ 不得修改 .git/config、remote、分支保护、CI 配置
  ✗ 不得触碰云端实例、Tailscale、阿里云 ECS 相关任何操作

  这两步永远由杨栋手动完成，或在杨栋明确逐次确认后才可执行。
  你完成本地验证后，只能「提示杨栋：本地已通过，建议提交」，然后停下等待。
```

```
【生产数据铁律】
  ✗ 不得读取、写入、删除 data/ 下任何真实数据文件（yd_os.db 等）
  ✓ 测试一律使用临时数据库（参照 tests/conftest.py 既有 fixture）
  云端实例承载真实飞轮数据，本地开发严禁以任何方式连接或影响云端数据。

  【手测 / 脚本强制项】
  任何 API 手测、脚本运行，必须先设 YD_OS_DB_PATH 指向临时库，
  严禁对 data/yd_os.db 执行 init_db() 或任何写操作。手测前先确认
  当前库路径不是 data/yd_os.db。
```

```
【既有系统铁律】（源自系统宪法 拾 · AI开发铁律）
  ✗ 不在 main 分支直接大改
  ✗ 不删除既有六模块（目标/任务/复盘/资产/能力/首页）
  ✗ 不破坏既有页面路由
  ✗ 不随意修改全局数据结构
  ✗ 不一次改太多功能
  ✗ 不跳过验收
```

## 1 · 开工前必须先答四问（系统宪法 捌，不得直接写代码）

收到任务后，进入 Plan Mode，先回答以下四问并产出 plan，等杨栋批准后才动手：

1. 推荐的实现顺序是什么，理由是什么？
2. 涉及哪些文件 / 新建哪些表，与既有 `database.init_db()` 如何整合？
3. 本地启动验证命令是什么？
4. 第一个 commit 后，如何验证基础框架跑通（页面能开、能存、刷新不丢）？

**默认全程使用 Plan Mode。重要改动先出 plan，杨栋签字后再执行。**

## 7 · 建议的本地验证 hook（可选，强烈推荐）

为强制执行修改路径铁律，建议在 `.grok/hooks/pre_tool.sh` 中拦截 push 类操作：

```bash
#!/usr/bin/env bash
# 拦截一切 push / 远程 / 部署命令，强制回到「杨栋手动」
case "$TOOL_COMMAND" in
  *"git push"*|*"git remote"*|*"ssh "*|*"scp "*|*"rsync "*|*deploy*)
    echo "⛔ 修改路径铁律：push/远程/部署 必须由杨栋手动执行。已拦截。"
    exit 1
    ;;
esac
exit 0
```

> 这道 hook 是机器层面的保险。即使 AI 一时忘了铁律，hook 也会把它挡在本地。
