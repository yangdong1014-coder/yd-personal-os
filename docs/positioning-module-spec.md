## 零 · 一句话定义

> 战略定位模块不是第七个执行模块，而是凌驾于「目标」之上的**方向校准层**。它回答「我此刻该往哪走、什么不该做」，并通过「AI 生成建议 → 人工确认」的方式，半自动地反向增删改目标系统。

---

## 壹 · 为什么要做这个模块（存在理由）

现有飞轮（目标 → 项目 → 任务 → 复盘 → 资产 → 能力）是**执行层闭环**：它保证「把事做好」，但保证不了「做的是对的事」。

现有「目标」模块只是目标的**容器**，不是目标的**校准器**。目标从哪来、对不对、之间怎么取舍——六模块没有任何一个回答。

战略定位模块补的就是这个上游缺口。它把飞轮从「执行闭环」升级为「执行 + 方向双闭环」：

```
        ┌─────────────────────────────────┐
        │   战略定位（方向线 · 校准层）     │
        │   定位锚 ──► 校准记录            │
        └──────────────┬──────────────────┘
                       │ 向下：半自动增删改目标（建议→确认）
                       ▼
目标 → 项目 → 任务 → 复盘 → 资产 → 能力
  │                                    │
  └────────── 向上：能力/资产数据反哺校准 ◄┘
```

**对宪法的补丁意义**：现有 14 条验收标准全是「建造验收」（能新建、能保存、刷新不丢），没有一条「运营验收」（系统是否真在产生对齐价值）。本模块的「目标对齐审查」就是第一条运营验收——把「飞轮转速」变成系统内的一等公民。

---

## 贰 · 设计第一性原则（本模块专属）

1. **触发机制 > 字段设计**：战略校准是低频、重思考的。字段再全，没有触发节奏就是死模块。本期不做定时强制（按用户决策：先手动，提醒后加），但数据结构要为「校准周期」留位，方便后续接提醒。
2. **校准的产出是动作，不是文字**：校准记录的价值不在写了多少反思，而在它**真实地增删改了哪些目标**。最后一环「目标变更」必须落到目标表的真实写操作，可追溯。
3. **半自动，人工确认**：复用 v1.11 智能归档 Inbox 的成熟范式——AI 解析/建议 → 人工预览勾选 → 入库。不静默改目标，不自动伪造关联。
4. **定位锚是不动点**：定位锚（第一性原理、身份内核、北极星）几乎不变，是校准时用来量一切的尺子。它和高频变动的校准记录必须分两层存储。
5. **服从宪法稳定优先**：不破坏既有六模块、不动既有路由、不改既有数据结构，只新增。

## 叁 · 数据结构

### 表一：positioning_anchor（定位锚 · 低频）

单行表（或最新一行有效），是当前生效的战略不动点。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| first_principle | TEXT | 第一性原理（如：研究并构建目标驱动、自迭代、结构稳定的系统） |
| identity_core | TEXT | 身份内核（如：创造者 + 提炼者，非表达者） |
| flywheel_def | TEXT | 飞轮定义（想/建模 → 造出来 → 被接受 → 成就感 → 想做更多） |
| current_stage | TEXT | 当前战略阶段：自我验证 / 外化企业 / 更广 |
| north_star | TEXT | 北极星指标：当前阶段唯一最重要的成功标准 |
| updated_at | TEXT | 更新时间 |

> 锚的「修订」本身是重大事件，每次修订建议保留历史（可用 updated_at + 软删除，或单独 anchor_history，第一版可只留最新一行 + 修改即覆盖，简单优先）。

### 表二：positioning_calibration（校准记录 · 周期性）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| calibrated_at | TEXT | 校准日期 |
| cycle | TEXT | 校准周期：月度 / 季度 / 触发式 |
| primary_contradiction | TEXT | 当前主要矛盾（此刻最大的张力） |
| doing_but_shouldnt | TEXT | 在做但不该做的（违背定位锚的消耗，待砍） |
| should_but_not_doing | TEXT | 该做但没做的（符合锚但被搁置的） |
| alignment_review | TEXT | 目标对齐审查结论（逐条审视当前主线目标是否仍对齐北极星） |
| conclusion | TEXT | 本期校准结论（下一步方向调整） |
| created_at | TEXT | |

### 表三：positioning_goal_action（校准触发的目标变更 · 半自动中间态）

这是本模块的杀手锏。校准记录产生的目标变更建议，先进此表作为 pending 中间态，人工确认后才真实写目标表。**直接复用 inbox 的 suggestion → commit 模式。**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| calibration_id | INTEGER FK | 来自哪次校准 → positioning_calibration.id |
| action_type | TEXT | 建议类型：新建目标 / 淘汰目标 / 降级目标 / 升级为主线 |
| target_goal_id | INTEGER | 涉及的既有目标 id（新建类为空） |
| payload | TEXT(JSON) | 新建目标的字段 / 变更内容 |
| reason | TEXT | 为什么建议这个变更（关联定位锚的哪一条） |
| status | TEXT | pending / confirmed / rejected |
| created_at | TEXT | |

**外键与级联**：参照 data-model.md 既有规范，calibration_id 外键级联；target_goal_id 软引用（目标可能已删，不强制级联删除本记录，保留审计痕迹）。

## 肆 · 接口（API）

沿用既有 Flask + JSON API 风格（参照 architecture.md / app.py 现有路由）。

```
# 定位锚
GET    /api/positioning/anchor              读取当前定位锚
PUT    /api/positioning/anchor              更新定位锚（重大动作，前端二次确认）

# 校准记录
GET    /api/positioning/calibrations        校准历史列表（时间倒序）
POST   /api/positioning/calibrations        新建一条校准记录
GET    /api/positioning/calibrations/<id>   单条详情（含其目标变更建议）

# 目标变更（半自动核心）
POST   /api/positioning/calibrations/<id>/suggest-actions
        # 可选 AI 辅助：把校准结论 + 当前主线目标 + 定位锚 喂给 AI，
        # 生成目标变更建议（新建/淘汰/降级），写入 positioning_goal_action 为 pending
POST   /api/positioning/actions/<id>/confirm   确认 → 真实写/改目标表
POST   /api/positioning/actions/<id>/reject    拒绝 → status=rejected
```

**confirm 的真实写操作规则**（严格遵守宪法 AI 铁律「不随意修改全局数据结构」）：
- 新建目标：在 goals 表插入一条，type 按 payload。
- 淘汰目标：**不硬删**，将目标状态改为「已淘汰/归档」（参照宪法旧系统处理原则：冻结归档而非删除）。
- 降级/升级：改 goals.type 字段（如 当前主线 ↔ 季度）。
- 所有 confirm 操作写入操作日志，可追溯「这个目标变更来自哪次校准」。

## 伍 · 前端

参照既有页面规范（templates/ + static/js/，原生 HTML/CSS/JS，无框架，复用 api.js / toast.js）。

```
templates/positioning.html      定位页
static/js/positioning.js        逻辑
```

页面分三区，对应三层逻辑：

1. **定位锚区（顶部，常驻）**：展示当前第一性原理、身份内核、北极星、当前阶段。克制、像「指挥部的作战方针」。可编辑（二次确认）。
2. **新建校准区**：填写主要矛盾 / 在做但不该做 / 该做但没做 / 目标对齐审查 / 结论。提交后可一键「生成目标变更建议」。
3. **目标变更确认区（复用 inbox 预览交互）**：列出 AI 生成或手填的 pending 变更建议，每条可勾选确认 / 拒绝。确认即真实改目标。下方接校准历史列表。

**UI 遵守宪法陆 · UI 规范**：深色背景 + 纸色文字，单色强调（暗橙），禁止彩虹标签/渐变/无意义动画。感受是「进入作战室」。

导航：在顶部导航条目标之前或之后，加一项「定位」（或「罗盘」）。

## 陆 · 提示词（若启用 AI 辅助）

新增提示词目录，沿用 prompts/<module>/<scene>.system.txt 规范，经 prompts/loader.py 加载：

```
prompts/positioning/suggest-actions.system.txt
prompts/positioning/suggest-actions.user.txt
```

**suggest-actions.system.txt 核心职责**：输入定位锚 + 本次校准结论 + 当前全部「当前主线」目标列表，输出结构化的目标变更建议（JSON），字段对齐 positioning_goal_action。要求：每条建议必须写明关联定位锚的哪一条（reason），不臆造目标，AI 只建议不执行。

## 柒 · 与既有系统的关系（迁移与兼容）

- **不动既有六模块的表结构。** 只新增三张表 + 一组路由 + 一个页面。
- **goals 表的写操作**：confirm 动作会写 goals 表。需确认 goals 表是否有「状态/归档」字段支持淘汰；若无，新增一个 status 字段（向后兼容，默认值为现状），这是唯一允许的对既有表的改动，且必须默认值兼容老数据。
- **导出兼容**：JSON 备份（database.py 导出）与 Obsidian zip（obsidian_export.py）需补入 positioning 三表。Obsidian 导出中，定位锚导为一张「战略定位.md」，校准记录每条导为一篇带 frontmatter 的 Markdown。
- **changelog**：完成后版本号按宪法升级规则走 x.0 → x+1.0（新增模块），更新 changelog.json 的 current 字段并打 git tag。

## 玖 · 验收检查表（本模块专属）

### 建造验收（功能层）

```
[ ] /positioning 页面能正常打开
[ ] 定位锚可读取、可编辑、刷新不丢
[ ] 能新建一条校准记录，所有字段可存
[ ] 校准历史列表按时间倒序展示
[ ] 能生成/手填目标变更建议（pending 态）
[ ] 确认建议后，目标表真实发生增/删/改
[ ] 拒绝建议后，status=rejected，目标表不变
[ ] 淘汰目标是归档而非硬删（数据可追溯）
[ ] 既有六模块未被破坏，原有数据未丢失
[ ] JSON 备份与 Obsidian 导出含 positioning 数据
[ ] 控制台无明显报错
[ ] 已输出修改日志、已 git commit、已更新 changelog
```

### 运营验收（价值层 · 本模块独有，是给宪法打的补丁）

```
[ ] 本模块上线后 30 天内，至少完成 1 次真实校准
[ ] 该次校准至少触发 1 个真实的目标变更（新建/淘汰/降级被确认）
[ ] 定位锚的北极星指标与当前主线目标可对应得上
```

> 运营验收不通过，说明模块虽建成但未真实运转——按宪法原则一「真实运转优先」，这才是模块成立的最终标准。
