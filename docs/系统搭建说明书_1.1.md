# 个人能力操作系统 · 系统搭建说明书 1.1

> 版本：1.1 · 2026.06
> 状态：✅ 现行有效
> 双用途：本人备忘 + AI执行上下文
> 原则：人读得懂逻辑，AI读得懂指令
> 宪法地位：本文件是系统最高约束文件，所有AI执行、迭代升级、模块开发均以此为准

## 零 · 系统宪法核心句

> 个人系统不是一堆工具，而是一套以目标为牵引、以任务为动作、以复盘为反馈、以资产为沉淀、以能力为结果、以AI为加速器、以版本机制为安全边界的长期进化系统。

一句话版本：

> **目标牵引，任务推进，复盘校正，资产沉淀，能力升级，AI加速，版本保命。**

---

## 壹 · 系统存在的理由

这个系统解决一个根本问题：

**我每天都在做事，但做完之后能力没有沉淀、资产没有积累、目标没有推进。**

忙碌和成长是两回事。这个系统要让两者挂钩——每一次执行都能转化为可见的资产和可量化的能力提升，最终驱动飞轮加速转动。

系统服务的不是"记录更多东西"，而是：

1. 更清晰地判断目标
2. 更稳定地推进任务
3. 更高效地复盘经验
4. 更系统地沉淀资产
5. 更持续地提升能力
6. 更主动地掌控人生与职业发展

系统的运转飞轮：

```
目标牵引
→ 项目拆解
→ 任务执行
→ 过程反馈
→ 复盘总结
→ 资产沉淀
→ 能力提升
→ 目标校正
→ 新一轮行动
```

这不是 to-do 工具，不是笔记软件，不是看板。
这是一套**以目标为驱动、以资产为产出、以能力为度量**的个人操作系统。

## 贰 · 系统架构总览

系统由三条并行的线构成：

```
主线（执行链）
目标 ──► 项目 ──► 任务 ──► 执行 ──► 复盘 ──► 资产 ──► 能力提升

治理线（规则链）
设计原则 ──► 操作规则 ──► 反馈机制 ──► 升级机制

支撑线（工具链）
本地系统 + AI协作 + Obsidian知识库 + Git版本控制

输入入口层（v1.11+）
非结构化输入 → 智能归档 Inbox → 预览确认 → 结构化资产
```

三条线同时运转，主线产出结果，治理线保持质量，支撑线提升速度。智能归档属于**输入入口层**：将零散文字经 AI 解析、人工确认后，转入目标/项目/任务/复盘/资产/能力等模块，不静默覆盖已有数据。

## 叁 · 第一性原则（七条）

### 原则一：真实运转优先

系统不是为了好看，不是为了复杂，不是为了演示，而是为了真实使用。

判断标准只有一个：
> 它是否能帮助我每天、每周、每月更稳定地推进关键目标？

不服务真实运转的功能，暂缓。

---

### 原则二：目标驱动优先

所有任务、项目、复盘、资产，都必须能追溯到目标。
系统的第一入口不是任务，而是目标。
没有目标牵引的任务，容易变成忙碌。
没有任务支撑的目标，容易变成口号。

---

### 原则三：闭环优先

系统不是信息堆积，而是闭环运转。最小闭环为：

```
目标 → 项目 → 任务 → 执行 → 复盘 → 资产 → 能力 → 目标校正
```

第一版必须优先跑通这个闭环，而不是把每个模块做厚。

---

### 原则四：稳定优先于复杂

建设顺序必须是：

```
先能跑 → 再稳定 → 再好用 → 再扩展 → 再智能 → 再产品化
```

任何会破坏系统稳定性的功能，都必须延后。

---

### 原则五：AI加速，但不能失控

AI是系统加速器，不是无边界接管者。

AI可以参与：目标拆解、任务生成、复盘提炼、资产沉淀、能力分析、页面开发、系统优化。

AI不可以：无边界修改系统、直接破坏稳定版本、跳过验收、随意重构核心数据结构、删除已有模块、在没有版本保护的情况下连续执行。

---

### 原则六：资产沉淀优先

判断一个任务有没有价值，不只看它是否完成，还要看它有没有留下可复用资产。

每一次重要行动，都要尽量转化为：
知识卡片 / SOP / 提示词 / 工作流 / 案例复盘 / 方法论 / 可复用模板

---

### 原则七：主动权优先

凡是进入系统的目标、项目和任务，都要问：
- 它是否提升我的主动权？
- 它是否沉淀我的个人资产？
- 它是否形成复合杠杆？
- 它是否能在6-12个月后产生复利？

不提升主动权的低价值消耗，要被识别、压缩或剔除。

## 肆 · 核心模块定义

### 模块 A · 目标系统

**存在目的：** 防止陷入点状忙碌，让所有行动能追溯到方向

**数据结构：**
```
目标（Goal）
  ├── 目标名称
  ├── 目标类型：年度 / 季度 / 月度 / 当前主线
  ├── 为什么重要
  ├── 对应主动权
  ├── 当前状态
  └── 项目（Project）× N
        └── 任务（Task）× N
```

**MVP完成标准：**
- 能新建目标并保存（含类型字段）
- 能在目标下新建项目
- 能在项目下新建任务，任务可改状态
- 刷新页面数据不丢失

**第一版不做：** 截止日历视图、任务依赖关系、多人协作

---

### 模块 B · 任务系统

**存在目的：** 服务日常执行，让每日推进可见

**数据结构：**
```
任务（Task）
  ├── 任务名称
  ├── 所属项目
  ├── 优先级：高 / 中 / 低
  ├── 执行状态：待处理 / 进行中 / 完成
  ├── 今日是否推进
  ├── 完成结果
  ├── 是否需要复盘
  └── 是否产生资产
```

**MVP完成标准：**
- 任务列表可见，状态可改
- 可标记"今日推进"
- 可标记"需要复盘"/"产生资产"

---

### 模块 C · 复盘系统

**存在目的：** 识别偏差，提炼经验，驱动资产沉淀

**数据结构：**
```
复盘（Review）
  ├── 复盘日期
  ├── 复盘类型：每日 / 每周 / 项目 / 事件
  ├── 今天做了什么
  ├── 推进了什么
  ├── 卡住了什么及原因
  ├── 下一步调整
  └── 可沉淀内容（触发资产生成）
```

**MVP完成标准：**
- 能新建一条复盘记录并保存所有字段
- 能看到复盘历史列表
- "可沉淀内容"字段填写后可一键跳转到资产模块新建卡片

---

### 模块 D · 资产系统

**存在目的：** 将每次执行和复盘转化为可复用的知识资产

**资产类型（第一版只做知识卡片）：**
```
知识卡片（KnowledgeCard）
  ├── 标题
  ├── 触发情境（这张卡片从什么事情来的）
  ├── 核心内容
  ├── 可复用模型名（可选）
  ├── 资产类型：知识卡片 / SOP / 提示词 / 工作流 / 案例复盘 / 方法论
  ├── 关联能力模块标签（多选）
  ├── 来源任务 / 来源复盘（可选）
  └── 创建日期
```

**MVP完成标准：**
- 能新建知识卡片并保存所有字段
- 能看到全部卡片列表（按时间倒序）
- 能按能力模块标签筛选
- 刷新后数据不丢失

**第二版再做：** 导出Markdown同步Obsidian、全文搜索、卡片关联任务

---

### 模块 E · 能力系统

**存在目的：** 让八个能力模块的成长可见、可记录、可回溯

**八个固定模块（不可增删，顺序固定）：**

| 序号 | 模块名 | 所属层级 | 核心定义 |
|------|--------|----------|----------|
| 01 | 本质力 | 基础认知层 | 穿透现象看本质的能力 |
| 02 | 建模力 | 基础认知层 | 将复杂问题抽象为可操作模型的能力 |
| 03 | 体系力 | 系统创造层 | 将零散认知组织为完整系统的能力 |
| 04 | 产品力 | 系统创造层 | 将想法转化为可交付产品的能力 |
| 05 | 审美力 | 系统创造层 | 对质量和品味的判断力 |
| 06 | 创造力 | 系统创造层 | 产生新连接、新方案的能力 |
| 07 | 落地力 | 结果放大层 | 将计划转化为可验收交付物的能力 |
| 08 | AI驾驭力 | 结果放大层 | 以AI为杠杆放大个人产出的能力 |

**每个模块的记录结构：**
```
进展记录（Entry）
  ├── 日期
  ├── 内容（本次训练/应用了什么）
  ├── 来源项目（可选）
  ├── 关联资产（可选）
  └── 层级判断：能力层 / 应用层
      （能力>应用 = 未充分使用；应用>能力 = 过度延伸）
```

**MVP完成标准：**
- 八模块首页全部可见
- 每个模块可新增一条进展记录
- 每个模块可查看历史记录列表
- 刷新后数据不丢失

---

### 模块 F · 首页总览

**存在目的：** 每次打开系统，3秒内知道最重要的事是什么

首页只回答三个问题：
```
我当前最重要的目标是什么？
我本周最重要的项目是什么？
我今天必须推进的任务是什么？
```

首页不追求炫酷，只追求**清晰、克制、高信息密度**。

**MVP完成标准：**
- 显示当前主线目标（取类型为"当前主线"的目标）
- 显示本周进行中的项目列表
- 显示今日标记为"推进"的任务列表

## 伍 · 技术规格

### 运行环境要求
- 本地优先：一条命令启动，浏览器访问
- 手机端可用：响应式布局，后续升级为 PWA
- 无需互联网连接即可正常运行
- 数据存本地，不上云

### 技术栈规范
```
后端：Python 3.x + Flask
数据：SQLite（单文件，便于备份和迁移；不用mock数据，直接接真实存储）
前端：原生 HTML / CSS / JavaScript（不引入前端框架）
启动：python app.py → 访问 http://localhost:5000
```

### 目录结构规范
```
personal-system-v2/
├── app.py                    # 主入口
├── database.py               # 数据库初始化与操作
├── requirements.txt          # 依赖列表
├── README.md
├── data/
│   └── yd_os.db             # SQLite 数据文件
├── static/
│   ├── css/
│   │   └── main.css
│   └── js/
│       └── main.js
└── templates/
    ├── base.html
    ├── index.html            # 首页总览
    ├── goals.html            # 目标系统
    ├── tasks.html            # 任务系统
    ├── reviews.html          # 复盘系统
    ├── assets.html           # 资产系统
    └── capabilities.html     # 能力系统
```

### Git 分支规范
```
main        稳定版本（只接受验收通过的合并）
dev         开发版本
feature/xxx 单功能开发分支
backup/日期  关键备份节点
```

### Git Commit 规范
```
init: 项目初始化
feat: [模块名]-[功能描述]
fix:  [描述修复内容]
refactor: [描述重构内容]
docs: [说明书或文档更新]
```

**铁律：不存档，不推进。**

## 陆 · UI 设计规范

### 核心感受
这是一个严肃的个人工具，不是消费级 App。
打开它的感受应该是：**进入指挥部，不是打开微博。**

参照感：Linear / Notion 的秩序感，但更个人、更克制。

### 视觉规范
```
主色调：深色背景（#1a1a1a 或接近）+ 纸色文字（#f0ece0）
强调色：单色，克制使用。推荐暗橙/暗红系（#c0603a 附近）
禁止：彩虹色标签、渐变大背景、卡片投影堆叠、无意义动画
字体：中文 Noto Sans SC，数据/代码 JetBrains Mono
```

### 交互规范
- 操作路径最短：能一步完成的不设两步
- 数据即时保存：不需要手动点"保存"按钮
- 空状态有引导：每个模块空白时显示"添加第一个xxx"
- 页面要让人一眼知道：现在是什么状态 / 最重要的事是什么 / 下一步该做什么

### 导航规范
```
顶部固定导航：
[系统名]  |  首页  目标  任务  复盘  资产  能力
```

## 捌 · 执行节奏与验收

### 标准执行流程

```
1. AI收到指令
2. AI说明本Step要做什么（不超过3条）
3. AI执行
4. AI完成后汇报：做了什么 + 验收方法
5. 我对照验收检查表验收
6. 验收通过 → git commit
7. AI提示下一步计划，等待确认
8. 确认后进入下一个Step
```

### 完整验收检查表

每次开发完成后，必须检查：

```
[ ] 页面能正常打开
[ ] 核心模块还在
[ ] 原有数据未丢失
[ ] 页面样式未被破坏
[ ] 控制台无明显报错
[ ] 新增功能符合目标
[ ] 没有影响其他模块
[ ] 已输出修改日志
[ ] 已完成 git commit
```

|---|------|--------|----------|

| 2 | A目标 | 拆解项目 | 目标下可添加项目 |

| 4 | B任务 | 任务列表 | 任务可见，状态可改 |

| 6 | C复盘 | 新建复盘 | 所有字段可填写保存 |
| 7 | C复盘 | 复盘历史 | 历史列表按时间倒序展示 |
| 8 | D资产 | 建卡片 | 所有字段可输入保存 |
| 9 | D资产 | 看列表 | 全部卡片按时间倒序展示 |

## 玖 · AI 协作规范

### 角色分工
```
杨栋            定方向、定判断、定取舍、人工验收
Claude / GPT    定结构、拆逻辑、方案评审、生成提示词
Codex / CLI     执行开发、修改代码、实现功能
Git             版本保护、记录变化、支持回滚
本说明书        最高约束文件
```

### AI开始前必须回答四个问题（不得直接写代码）
1. 推荐的技术栈是什么，理由是什么？
2. 项目目录结构长什么样？
3. 本地启动命令是什么？
4. 第一个commit之后，如何验证基础框架跑通了？

### 每次AI修改前必须明确
```
本次修改目标
允许修改范围
禁止修改范围
涉及模块
验收标准
是否需要新建分支
是否允许影响已有功能
```

### 每次AI修改后必须输出
```
修改了哪些文件
每个文件修改了什么
为什么这么改
影响了哪些模块
如何测试
是否存在风险
是否建议提交版本
下一步建议
```

### Step完成后的标准汇报格式
```
✅ Step X 完成
修改文件：[列表]
建议执行：git commit -m "feat: xxx"
下一步：[说明下一步计划]
等待确认后继续。
```

## 拾 · AI开发铁律与系统禁令

### AI开发铁律
```
不允许直接在 main 分支大改
不允许无说明重构核心结构
不允许删除已有模块
不允许破坏已有页面路由
不允许随意修改全局数据结构
不允许一次性改太多功能
不允许跳过验收
不允许没有 commit 就继续下一轮大改
```

### 系统最高禁令
```
为了酷而增加功能
为了完整而牺牲稳定
为了速度跳过版本管理
为了自动化放弃人工验收
为了展示做无用复杂度
为了工具而工具
为了记录而记录
为了大而全破坏最小闭环
```

### 每次新增功能前必须问
```
这个功能是否服务目标？
是否提升执行？
是否帮助复盘？
是否促进资产沉淀？
是否提升能力？
是否增强主动权？
是否值得现在做？
是否会破坏系统稳定性？
```

如果答案不清楚，暂缓。

| 变更类型 | 版本变化 | 示例 |
